## Context

`workout_log` is a Flutter mobile app on Dart SDK `>=3.8.0 <4.0.0`. The current `pubspec.yaml` was last touched at version `1.2.3+8` and most direct dependencies are now 1‚Äì2 minor versions behind. The dev-dep codegen chain (`build_runner` 2.6.0 ‚Üí 2.15.0, `json_serializable` 6.10.0 ‚Üí 6.14.0) is far enough behind that its own transitive deps (`build_resolvers`, `build_runner_core`) are marked **discontinued** by `flutter pub outdated`. Three downstream proposals (DB+freezed, Riverpod, UI refactor) all need newer codegen.

`flutter analyze` currently reports 12 warnings ‚Äî a mix of `unnecessary_null_comparison` (dead null-checks left over from pre-null-safety code), `dead_code` (unreachable returns after exhaustive enum switches), one `deprecated_member_use` (`textScaleFactor`), and one `unnecessary_set_literal`. None are runtime bugs today but each is a future bug-magnet and they collectively make the warning baseline noisy.

`Uuid().v1()` is used to generate IDs for `WorkLog` and `Exercise`. v1 embeds the host's MAC address and timestamp in the ID. This is a fingerprinting vector that has no upside for an offline workout app.

## Goals / Non-Goals

**Goals:**
- Every direct + dev dep at latest resolvable (no `*` markers in `flutter pub outdated` for the rows we own).
- `flutter analyze` reports zero warnings, zero infos.
- Entity IDs generated with `Uuid().v4()`. Existing on-device v1 IDs continue to work (IDs are opaque strings).
- App boots on Android emulator and the smoke flow (open ‚Üí swipe-delete a workout ‚Üí add an exercise ‚Üí toggle theme) works unchanged.

**Non-Goals:**
- No behavior changes, no UI changes, no API/DB schema changes.
- No file renames (kept for proposal #2 + #4).
- No state-management migration (proposal #3).
- No new features.

## Decisions

### D1 ‚Äî Bump all deps in one commit, not staged

**Decision**: One pubspec edit covers every bump (direct + dev + uuid switch), one regen of `*.g.dart`, one analyze sweep.

**Rationale**: The deps form a graph. Bumping `json_serializable` without `build_runner` doesn't resolve. Bumping `flutter_slidable` independently is the only one with API-surface risk and gets its own validation step. Staging into 5 commits would force re-running `pub get` + regen between each ‚Äî pointless churn for a mechanical pass.

**Alternative considered**: One commit per dep. Rejected ‚Äî codegen chain bumps are coupled, and the user explicitly asked for larger proposals.

### D2 ‚Äî `flutter_slidable` 3 ‚Üí 4 verification approach

**Decision**: Smoke-test the swipe-delete UI in `workLogPageView.dart` manually on the Android emulator after the bump. No automated test (tests don't exist yet ‚Äî proposal #4 introduces them).

**Rationale**: `flutter_slidable` 4.x kept `Slidable`, `ActionPane`, `SlidableAction`, `ScrollMotion`, `extentRatio` API-compatible. The only realistic break is `motion:` or `key:` semantics, which a 30-second emulator run catches.

**Alternative considered**: Add a widget test as part of this proposal. Rejected ‚Äî testing infrastructure belongs in proposal #4's scope; mixing in a single test here would set a precedent we don't want.

### D3 ‚Äî `Uuid().v4()` migration is forward-only

**Decision**: New IDs generated by `v4()`. Existing v1 IDs in the user's local sqflite DB are left as-is ‚Äî they remain valid string PKs.

**Rationale**: IDs are opaque strings (`VARCHAR(32)` PK). The migration is purely "stop generating new v1 IDs". No data migration required.

### D4 ‚Äî Fix `flutter analyze` warnings as part of this proposal, not deferred

**Decision**: All 12 warnings fixed here.

**Rationale**: Per global rule #1 ("do the good job"). The warnings are 5-minute fixes individually; deferring them sets a baseline of "we have warnings" that everyone will then tolerate. Zero-warning baseline makes proposal #2's analyzer output meaningful.

**Sub-decisions:**
- `unnecessary_null_comparison` ‚Üí delete the dead `if (x == null)` branch entirely (not silenced with `// ignore:`).
- `dead_code` `return Colors.white70` / `return ""` after exhaustive enum switches ‚Üí delete; the switch is already exhaustive.
- `deprecated_member_use` `textScaleFactor: 3` ‚Üí migrate to `textScaler: TextScaler.linear(3)`.
- `unused_local_variable` `log` in `dbProvider.dart:228` ‚Üí delete (the `WorkLog log;` declaration is unused).
- `unnecessary_set_literal` `{sum += int.parse(v)}` ‚Üí unwrap to `sum += int.parse(v)`.

### D5 ‚Äî `flutter_launcher_icons` 0.13 ‚Üí 0.14 config

**Decision**: Keep the `flutter_icons:` block in pubspec.yaml; 0.14 still accepts it (the new preferred key is `flutter_launcher_icons:` but both work).

**Rationale**: Minimum-diff. If 0.14 starts warning, switch the key in a follow-up.

## Risks / Trade-offs

- **[Risk]** `flutter_slidable` 4.x surface-level API break in `_createWorkLogRowWidget` ‚Üí **Mitigation**: smoke-test swipe-to-delete manually post-bump; revert just that dep if broken.
- **[Risk]** Codegen output churn ‚Äî `*.g.dart` files will diff substantially after `json_serializable` 6.10 ‚Üí 6.14 ‚Üí **Mitigation**: expected; reviewed-once, committed as part of this proposal. Generated files are not behavior changes.
- **[Risk]** `dart run build_runner build --delete-conflicting-outputs` deletes legitimate generated files ‚Üí **Mitigation**: only `entity/*.g.dart` are generated; verify git status post-regen shows only the two expected files changed.
- **[Risk]** Hidden runtime regression from a transitive bump ‚Üí **Mitigation**: app smoke flow on emulator before declaring done.

## Migration Plan

1. Edit `pubspec.yaml` (direct + dev + transitive overrides if needed).
2. `flutter pub get`.
3. `dart run build_runner build --delete-conflicting-outputs`.
4. Fix the 12 analyzer warnings (5 files touched).
5. Switch `Uuid().v1()` ‚Üí `Uuid().v4()` (2 files).
6. `flutter analyze` ‚Üí must be 0 issues.
7. `flutter run -d <android-emulator>` ‚Üí smoke flow.
8. Commit (single commit, backdated per global rule).

**Rollback**: `git revert <sha>`; deps revert to prior `pubspec.yaml`.
